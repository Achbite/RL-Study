
# AI 编码助手工作规范

你在编写和修改代码、文档时，必须严格遵循以下规范。本规范适用于所有开发场景，具体业务上下文会在工程开始前另行说明。

---

## 一、工作模式

### 默认模式：方案规划审核模式

除非用户明确说"直接实现"或"落地"或"执行"，否则一律进入方案规划审核模式：

- 只输出方案设计、架构分析、代码草案或文档规划
- 不实际修改任何代码文件
- 不实际创建任何新文件
- 等待用户明确确认后，才进行实际的代码落地修改

### 代码修改许可规则

- 不允许修改或删除用户需求未涉及的代码块
- 不允许随意新建文件，除非是开发一个全新功能块所必须的，且已在方案中说明并得到用户许可
- 用户手动注释掉的代码保留不动，除非用户明确要求清理该功能
- 涉及多个文件的修改，需在方案中列出变更清单，经确认后再逐一执行
- 如果发现现有代码有问题但不在本次需求范围内，可以在方案中提出但不直接修改

### 编译与验证规则

- 不允许直接运行终端进行编译检验（如 bazel build 等），因为涉及环境问题编译检查容易出错
- 由用户自行在对应环境中进行编译验证，然后把结果反馈
- 除非用户明确指示，不需要主动去验证或执行验证流程

### 临时检查文件规则

当需要通过脚本检查日志、分析信息或执行诊断命令时，为避免控制台输出读取问题（截断、编码错误、管道阻塞等），允许使用临时检查文件：

-**使用场景**：仅在需要通过终端执行脚本/命令并获取输出结果时使用，不用于其他目的

-**文件命名**：统一命名为 `_check_output.txt`，固定使用此文件名

-**存放位置**：固定放在项目根目录下（即工作区根目录）

-**使用方式**：将脚本/命令的输出重定向到该文件（如 `> _check_output.txt`），然后通过 read_file 读取结果

-**一次对话一个文件**：每次对话仅使用同一个临时文件，后续检查直接覆盖写入（`>` 覆盖而非 `>>` 追加），不创建多个临时文件

-**及时清理**：在当前检查任务完成、结果已读取并分析后，必须立即删除该临时文件

-**不纳入版本控制**：确保项目的忽略文件（`.gitignore` / `.ignore` 等）中已包含 `_check_output.txt`，防止误提交；如未包含，在首次使用时提醒用户添加

-**红线豁免**：临时检查文件的创建和删除不需要用户确认（它是 Agent 的临时工作文件，不属于项目代码或文档），但其内容仅用于读取分析，不得作为项目产出物保留

### 技术文档生成规则

- 未经用户许可，不可随意生成技术方案文档
- 生成或修改文档时，必须按章节分段写入，避免文本过长导致写入失败
- 已有的技术文档作为参考，新文档的格式风格应保持一致

### 技术文档维护规则

- 文档中的"状态更新"（如完成状态标记、验证结果打勾等）可以直接写入文档
- 文档中的"方案更新"（如新功能方案、架构变更等）需要用户审核确认后才能写入
- 文档中的代码示例和伪代码也必须遵循本规范中的注释风格

### 🔴 红线规则（任何情况下不可违反）

> **任何代码文件的新建或修改，以及任何文档的新建或内容变更，都必须先输出方案并获得用户明确同意后才能执行。没有例外。**

具体包括：

- 新建任何文件（代码文件、配置文件、文档文件、脚本文件等）：必须先说明文件用途、存放位置、内容概要，获得用户同意
- 修改任何现有文件：必须先说明修改原因、变更内容摘要、影响范围，获得用户同意
- 唯一例外：文档中的纯状态更新（打勾、标记完成等）可以直接执行，但内容性变更仍需确认
- 唯一例外：临时检查文件（`_check_output.txt`）的创建和删除可以直接执行（见"临时检查文件规则"）

## 二、C++ 注释与格式规范

### 2.1 函数注册区（Register）注释规范

按从属分类组织，每个分类用单行注释标注类别名，注册行末尾用行内注释写明含义。

格式要求：

- 分类名简洁，按功能属性归类
- 行内注释格式：// 中文名称：一句话功能描述
- 所有行内注释尽量对齐到同一列
- 不同分类之间空一行

示例：

```cpp

    // 稀疏奖励

    RegisterOnce("PassReward", PassReward);           // 通关奖励：成功到达目标时给予的大额奖励

    RegisterOnce("TimeBonus", TimeBonus);             // 通关时间奖励：通关时间越短奖励越高；未通关按进度分级


    // 密集奖励（每帧，动态系数）

    RegisterOnce("DistanceReward", DistanceReward);   // 距离引导：靠近目标时给予正向奖励，远离时惩罚（含探索奖励）


    // 事件驱动

    RegisterOnce("FallPenalty", FallPenalty);          // 坠落惩罚：每次坠落时给予惩罚

```

### 2.2 函数定义（实现区）注释规范

所有函数定义前统一使用 `// ---- 函数中文名 ----` 轻量单行分隔，不论函数复杂度。

格式要求：

- 一行写完函数中文名，简要概括该函数的功能
- 分隔线与函数定义之间不空行
- 不展开子机制描述（详细的多阶段/多规则说明放到项目文档中）
- 不写变更说明、版本标记、参数说明、返回值说明

示例：

```cpp

    // ---- 坐标范围检测（地图边界）----

    staticboolIsCoordInRange(floatx, floaty, floatz) {


    // ---- 三维距离计算 ----

    staticfloatDistance3D(float x1, float y1, float z1, float x2, float y2, float z2) {


    // ---- waypoints 校验 ----

    NaviCheckResult CheckWaypoints(constraid::AgentState& agent_state, int agent_id, bool mute) {


    // ---- NaviCheckMonitor::Tick（持续监控入口）----

    voidNaviCheckMonitor::Tick(constraid::AgentState& agent_state, int agent_id) {


    // ---- 卡住检测 ----

    voidNaviCheckMonitor::CheckStuck(AgentNaviState& s, constraid::AgentState& agent_state, int agent_id) {

```

### 2.3 头文件函数声明注释规范

头文件中的函数声明不使用分隔线，采用注册区风格：按分类用单行注释标注类别名，声明行末尾用行内注释写明功能。

格式要求：

- 分类注释标注类别，声明行末尾行内注释简要描述功能
- 行内注释尽量对齐到同一列
- 不同分类之间空一行
- 类声明使用 `// ---- 类中文名（补充说明）----` 一行分隔

示例：

```cpp

    // 静态校验

    NaviCheckResultCheckWaypoints(...);          // waypoints 校验

    NaviCheckResultCheckTargetLocation(...);     // target_location 校验

    NaviCheckResultCheckReachableDistance(...);   // reachable_distance 校验

    NaviCheckResultCheckNaviData(...);           // 综合校验入口


    // ---- 持续监控器（跨帧状态追踪，每个 Agent 独立）----

    classNaviCheckMonitor {

    public:

        voidTick(...);


    private:

        // 时序校验

        voidCheckStuck(...);           // 卡住检测

        voidCheckTargetJump(...);      // 目标跳变检测

        voidCheckWaypointsEmpty(...);  // waypoints 持续为空检测

        voidCheckDrifting(...);        // 远离目标检测

    };

```

### 2.4 常量定义区注释规范

用 `// --- 分类名 ---` 分组，每个常量行末写行内注释。

格式要求：

- 分类名用 `--- xxx ---` 格式
- 行内注释对齐到同一列
- 相关常量紧邻排列，不同分类之间空一行
- 如果某个常量有额外约束或概念说明，可在该组常量上方用单行注释补充

示例：

```cpp

    // --- 距离引导 ---

    constexprfloat kDistCoef_Min = 1.0f;               // 距离引导最小系数

    constexprfloat kDistCoef_Max = 6.0f;               // 距离引导最大系数（停滞时boost到此值）

    constexprfloat kDistCoef_BoostStartFrames = 60;    // 多少帧停滞后开始boost

    constexprfloat kDistCoef_BoostRampFrames = 120.0f; // 从min到max的过渡帧数


    // --- 温度参数 ---

    // 概念：类似模型推理的temperature，控制信号的"锐度"

    // 高温度 → 差异小 → 更愿意探索不同策略

    // 低温度 → 差异大 → 更倾向利用已知最优策略

    constexprfloat kTemp_Default = 1.0f;                // 默认温度

    constexprfloat kTemp_Min = 0.5f;                    // 最低温度（最收敛）

```

### 2.5 函数体内注释规范

关键逻辑步骤前用单行注释说明意图。

格式要求：

- 写做什么、为什么，不解释语法
- 如果函数体内有多个阶段，用 `// ---- N. 阶段名 ----` 分隔
- 分支判断前简要说明分支条件的含义
- 不为每一行都写注释，只在关键逻辑处写

阶段分隔示例：

```cpp

    // ---- 1. 初始化上下文 ----

    InitContext(params);


    // ---- 2. 执行核心逻辑 ----

    ProcessCore(data);

```

逻辑注释示例：

```cpp

    // 正弦波动：让温度随时间自然起伏，帮助跳出局部最优

    float wave = amplitude * std::sin(...);


    // 止损修正：回撤过大时收紧参数

    if (is_drawdown_triggered) {

```

分支注释示例：

```cpp

    // --- 规则1：前方空旷 → 正奖励 ---

    if (front_dist > threshold) {


    // --- 规则2：前方被堵 → 惩罚，但距离在缩短时减轻 ---

    if (front_dist < min_threshold) {

```

### 2.6 代码清理规范

- 用户手动注释掉的代码保留不动，除非用户明确要求清理该废弃功能
- 当用户明确要求清理某个废弃功能时，才彻底删除该功能的所有相关代码（函数声明、函数体、注册行、相关常量），不留残余注释
- 不保留 "已废弃"、"旧方案"、"弃用" 等标注性注释
- 不写变更日志类注释（如 "// 2024-01-01 删除了xxx"、"// 新增了xxx"、"// 阶段二新增"）
- 不写 "采用xx方案替代"、"改进了xx" 之类的方案对比或变更说明注释
- 所有注释只描述当前功能的作用和意图，不描述历史演变

## 三、通用代码生成规范

### 3.1 注释语言与风格

- 注释语言统一使用中文
- 注释描述的是功能作用和设计意图，不是代码翻译
- 每个新增的参数、变量、代码块都必须有对应注释，按上述分区规范书写
- 不同类别/阶段的代码必须分块，用对应的分隔格式隔开
- 保持同一文件内注释风格完全一致

### 3.2 代码修改原则

- 只修改用户需求明确涉及的部分，不做"顺手优化"
- 修改前必须理解上下文，不破坏已有逻辑和代码结构
- 新增代码的风格必须与所在文件现有风格保持一致
- 如果发现现有代码有问题但不在本次需求范围内，可以在方案中提出但不直接修改
- 涉及多文件修改时，先列出完整的变更清单（文件名 + 修改内容摘要），经确认后再执行

### 3.3 风格一致性要求

- 生成的代码必须与目标文件的现有风格保持一致，包括但不限于：命名风格、缩进方式、大括号位置、空行习惯、注释风格
- 插入新代码前，先观察周围代码的写法，严格模仿同文件已有的模式
- 不同文件之间风格可能不同，以目标文件为准，不跨文件统一风格
- 如果同一文件内风格本身就不一致，以该文件中占多数的风格为准

### 3.4 文件操作约束

- 优先编辑现有文件，除非有充分理由才新建文件
- 新建文件必须在方案中说明原因和文件位置，经用户确认后才创建
- 不随意移动、重命名现有文件
- 不删除任何文件，除非用户明确要求

### 3.5 无效修改禁止

- 不做纯格式调整（如只改空行、只调缩进、只改注释措辞）而不改功能逻辑的修改
- 不做等价重写（如把 if-else 改成三目运算符、把 for 改成 range-based for）而不改功能逻辑的修改
- 不做"顺手重构"（如重命名变量、提取子函数、调整函数顺序）除非用户明确要求
- 每次修改必须有明确的功能目的，能回答"这次改动解决了什么问题"

## 四、Markdown 文档排版规范

### 4.1 数学公式

- 行内公式使用 `$...$` 包裹，公式与左右文字之间用空格隔开

  - 正确：向量 $\mathbf{x}$ 表示输入
  - 错误：向量$\mathbf{x}$表示输入
- 块级公式使用独立行 `$$` 包裹，公式前后各留一个空行
- 使用标准 LaTeX 语法
- 分段公式使用 `\begin{cases}...\end{cases}` 或 `\begin{aligned}...\end{aligned}`

### 4.2 表格

- 使用标准 Markdown 管道表格
- 表格前使用加粗描述性标题引出，格式：**标题名**：
- 表格前后各留一个空行
- 表格内容简洁，避免单元格内放过长文本

### 4.3 流程图（Mermaid）

- 使用 ` ```mermaid ` 代码块包裹
- 节点文字使用 `["..."]` 方括号+引号包裹
- 箭头标签使用 `|"..."|` 竖线+引号包裹

### 4.4 其他排版细则

- 一级标题用于文档总标题，二级标题用于大章节，逐级递进
- 引用块用于章节定位说明、核心提醒、前置条件等元信息
- 代码块标注语言类型
- 章节之间使用 `---` 水平分隔线隔开（仅用于大章节之间）

## 五、文档编写与维护规范

### 5.1 写入策略

- 大型文档必须按章节分段写入，单次写入不超过一个主章节
- 写入顺序从前到后，避免插入式修改导致偏移
- 每段写入后确认成功再继续下一段

### 5.2 文档修改原则

- 只修改需求涉及的章节，不动其他章节
- 保持与项目已有技术文档的格式风格一致
- 章节内容变更时同步更新目录（如有）

### 5.3 文档维护双轨制

- 状态更新（完成标记、验证结果打勾、日期更新等）：可直接写入文档，无需等待确认
- 方案更新（新功能设计、架构变更、新增章节等）：先输出方案供用户审核，确认后再写入文档
- 文档中引用的代码示例也必须遵循第二章的 C++ 注释规范

## 六、Skill 知识库系统

> 核心理念：项目的技术文档即 Agent 的 Skill 知识库。每次接手任务前先学习相关 Skill，完成任务后主动提议更新 Skill 文档，形成持续进化的项目记忆。本机制不依赖特定项目，适用于任何具有技术文档目录的工程。

### 6.1 Skill 文档定义与结构

Skill 文档是以 Markdown 格式存放在项目指定目录下的技术参考文档，记录了项目的架构设计、模块职责、协议定义、配置说明、运维操作等关键知识。

#### 标准 Skill 文档结构

每份 Skill 文档采用以下标准结构，借鉴 Claude Code Skill 的三级加载体系：

```

skill-document/

├── SKILL.md (文档正文)          - Agent 使用的核心知识文档

│   ├── Frontmatter (YAML)      - name + description（元数据，始终可见）

│   └── Body (Markdown)         - 正文内容（按需加载）

├── README.md (可选)             - 人类可读的维护说明

└── 捆绑资源 (可选)

    ├── scripts/                 - 辅助脚本（自动化操作、数据处理等）

    ├── references/              - 补充参考文档（按需加载到上下文）

    └── assets/                  - 模板、配置示例等资源文件

```

#### Frontmatter 规范

每份 Skill 文档必须包含 YAML Frontmatter：

```yaml

---

name: skill-document-name          # 英文标识名，kebab-case

description: 一句话描述文档用途和适用场景。当涉及 XXX 模块的开发、调试或维护时应加载此文档。

metadata:

  author: Team Name                # 维护团队/作者

  version: "1.0"                   # 文档版本

  depends_on:                      # 关联的其他 Skill 文档（可选）

    - other-skill-doc.md

---

```

**关键规则**：`description` 字段是 Agent 判断何时加载文档的唯一依据，必须清晰描述文档覆盖的知识域和触发场景。

#### 渐进式披露（三级加载）

Skill 文档使用三级加载系统高效管理上下文：

| 级别 | 内容 | 加载时机 | 大小限制 |

|------|------|---------|---------|

| L1：元数据 | Frontmatter（name + description） | 始终在上下文中 | ~100 词 |

| L2：正文 | SKILL.md / 文档正文 | Skill 被触发时加载 | 建议 < 500 行 |

| L3：资源 | scripts/ / references/ / assets/ | Agent 按需读取 | 无硬限制 |

**默认假设：Agent 已经非常聪明。** 只在文档中添加 Agent 还不具备的项目特定上下文。质疑每段信息："Agent 真的需要这个解释吗？"

### 6.2 Skill 发现流程（每次任务开始前执行）

> **强制规则**：Agent 在接到开发任务后，必须主动使用工具搜索和加载相关 Skill 文档。这是自主行为，无需等待用户指示"去看文档"。搜索和读取 Skill 文档不受红线规则约束（属于只读操作）。

接到开发任务后，按以下流程加载相关 Skill 知识：

**第零步：自主评估（是否需要加载文档）**

- 收到任务后，先判断任务是否涉及项目特定知识（模块架构、协议字段、配置规格等）
- 如果任务足够简单且不涉及项目特定知识（如纯语法问题、通用算法），可跳过文档加载
- 如果不确定，宁可多搜索一次，也不要在缺乏上下文的情况下盲目开发

**第一步：扫描 Skill 目录**

- 使用工具列出 `.codebuddy/skills/` 目录下的所有子目录和 SKILL.md 文件
- 读取每份 SKILL.md 的 Frontmatter（前 10-20 行），获取 name 和 description
- 根据 description 与当前任务的关键词匹配，判断相关性
- 如果没有任何文档匹配当前任务，记录此信息，在方案中评估是否需要新建文档

**第二步：按需加载（L2 正文）**

- 只读取与当前任务直接相关的文档章节，不盲目读取全部文档
- 如果任务涉及多个模块的交互（如跨端通信、前后端联调），同时读取所有相关模块的文档
- 大文件优先读取目录结构，再定向读取相关章节

**第三步：按需加载（L3 资源）**

- 仅在需要执行特定操作时加载 scripts/
- 仅在需要深入了解某个子主题时加载 references/
- 不主动加载 assets/（仅在输出中引用时使用）

**第四步：知识校验**

- 在修改代码前，将文档中的描述与实际代码对照
- 如果发现文档与代码不一致，以代码为准进行开发，并在任务完成后提议更新文档

**加载决策参考**：

| 任务类型 | 应加载的 Skill 文档 |

|---------|-------------------|

| 修改某个子系统的代码 | 该子系统的项目文档 |

| 修改通信协议或跨端逻辑 | 所有涉及端的项目文档（确保协议/维度/字段一致） |

| 添加新功能模块 | 相关子系统文档 + 整体架构文档（了解模块间关系） |

| 排障或运维操作 | 操作手册 / 运维文档 |

| 修改专项功能（如导航、奖励等） | 对应的专项方案文档 |

### 6.3 Skill 引用规范

在方案设计和代码修改中引用 Skill 文档时：

- 方案中涉及架构决策时，引用对应文档的章节，例如："参考「XXX 项目文档 §4.2 核心消息结构」"
- 当文档信息与实际代码不一致时，以代码为准，并在方案中标注："文档 XXX §N.N 需同步更新"
- 不在代码注释中引用文档（代码注释只描述功能意图，不引用外部文档）

### 6.4 Skill 维护规范（代码变更后同步更新知识库）

> ⚠️ Skill 文档的维护受红线规则约束：所有文档的新建和内容变更，都必须先给出更新方案，获得用户确认后才能执行。

每次代码修改完成后，主动评估是否需要更新对应的 Skill 文档：

**触发更新评估的场景**：

- 新增、修改或删除了协议字段
- 新增、修改或删除了特征维度 / 动作空间 / 状态空间
- 新增、修改或删除了配置项或常量
- 新增了功能模块或子系统
- 修改了模块间的交互逻辑
- 修改了运维操作流程或命令

**更新流程**：

1. 在代码修改完成后，列出需要同步更新的文档清单（文档名 + 涉及章节 + 更新内容摘要）
2. 将更新清单作为方案的一部分输出，等待用户确认
3. 用户确认后，按章节逐一更新文档，遵循第五章的文档编写规范

**不更新的情况**：

- 纯粹的 bug 修复，不涉及接口/协议/配置变更
- 内部实现优化，不改变模块的外部行为
- 用户明确表示"不需要更新文档"

### 6.5 新增 Skill 文档规范

当开发了一个全新的功能模块，且现有文档无法覆盖时，可以提议新增 Skill 文档。

#### 新增前必须满足

- 现有文档确实无法通过新增章节来覆盖这部分知识
- 在方案中说明新文档的用途、定位、与现有文档的关系
- 获得用户明确同意

#### 自动提议触发条件

Agent 在以下情况应主动提议新建 Skill 文档（无需用户先提出）：

- 执行 §6.2 扫描后，没有任何现有文档的 description 能匹配当前任务涉及的模块
- 当前任务属于全新功能开发（新子系统、新协议、新模块间交互）
- 任务复杂度较高，涉及的项目特定知识（配置规格、协议字段、交互流程等）值得沉淀
- 提议时应在方案中简要说明：建议的文档名称、存放位置、应覆盖的知识域、建议的章节大纲

#### 新增文档流程

借鉴 Skill Creator 的标准创建流程，简化为以下步骤：

**步骤 1：明确文档定位**

- 确定文档覆盖的知识域（某个子系统？某个操作流程？某个专项功能？）
- 确定文档与现有文档的关系（独立？补充？关联？）
- 确定目标读者（Agent 自动加载？人类开发者参考？两者兼顾？）

**步骤 2：设计文档结构**

- 规划 Frontmatter（name、description、depends_on）
- 规划正文章节（按主题组织，每章聚焦一个方面）
- 评估是否需要捆绑资源（scripts / references / assets）

**步骤 3：输出方案并等待确认**

- 列出文件名、存放位置、章节大纲、内容概要
- 明确说明与现有文档的关系和分工
- 获得用户明确同意后才创建

**步骤 4：分段写入**

- 按第五章的文档编写规范，按章节分段写入
- 先写 Frontmatter + 概述，确认成功后再写后续章节

#### 新增文档要求

-**文件位置**：统一放在项目约定的技术文档目录下

-**文件命名**：简洁明确，反映文档主题（中文或英文均可，与现有文档风格一致）

-**Frontmatter**：必须包含 name、description 字段

-**文档格式**：遵循本规范第四章的 Markdown 排版规范

-**文档结构**：必须包含标题、概述、正文章节，按主题组织

#### 评估是否需要捆绑资源

| 资源类型 | 何时需要 | 示例 |

|---------|---------|------|

| `scripts/` | 存在需要确定性可靠执行的重复操作 | 数据处理脚本、格式转换工具、自动化部署脚本 |

| `references/` | 正文过长需要拆分，或有按需加载的补充资料 | API 文档、协议详细规格、算法推导 |

| `assets/` | 输出中需要引用的文件 | 模板文件、配置示例、图标资源 |

### 6.6 Skill 文档质量标准

Skill 文档应满足以下质量要求，以确保对 Agent 和开发者都有参考价值：

-**准确性**：文档内容必须与当前代码实际情况一致

-**完整性**：覆盖模块的核心接口、关键常量、配置项、交互协议等关键信息

-**简洁性**：只记录对开发决策有用的信息，不堆砌实现细节；优先使用简洁的示例而非冗长的解释

-**可维护性**：章节划分清晰，便于定向更新而不影响其他章节

-**无历史负担**：只描述当前状态，不描述历史演变（与代码注释规范一致）

-**自由度适配**：根据操作的脆弱性设置文档的具体程度

- 高自由度（文本指引）：适用于多种方法都有效的场景
- 中等自由度（伪代码/带参数脚本）：适用于有首选模式但允许变化的场景
- 低自由度（精确脚本/步骤）：适用于操作脆弱、一致性至关重要的场景

### 6.7 Skill 文档迭代维护

Skill 文档不是一次性产物，而是随项目持续进化的活文档。

**迭代触发时机**：

- 每次代码变更后的更新评估（见 §6.4）
- 用户在使用中发现文档不准确或缺失信息
- 项目架构发生重大变更
- Agent 在开发过程中发现文档与代码不一致

**迭代流程**：

1. 识别需要更新的内容（哪个章节、什么变化）
2. 输出更新方案（受红线规则约束）
3. 用户确认后执行更新
4. 更新 Frontmatter 中的 version（如有重大变更）

### 6.8 自主搜索与知识获取规范

Agent 在开发过程中应主动搜索和利用项目知识库，而非被动等待用户提供信息。

#### 项目 Skill 文档目录

本项目的 Skill 文档统一存放在：`.codebuddy/skills/` 目录下，每个 Skill 为一个子目录，核心文件为 `SKILL.md`。

#### 自主搜索触发规则

Agent 在以下场景必须主动搜索文档，无需用户指示：

-**接到新任务时**：按 §6.2 流程扫描 Skill 目录，定位相关文档

-**遇到不熟悉的模块时**：通过搜索工具在代码注释和文档中搜索关键词，建立上下文

-**跨模块交互时**：主动加载所有相关端/模块的文档，确保接口、协议、字段一致

-**修改配置或协议时**：搜索是否有对应的 Skill 文档记录了当前配置规格

-**任务执行中遇到歧义时**：优先搜索文档和代码获取答案，而非立即询问用户

#### 搜索方式优先级

1.**Skill 目录扫描**：优先扫描 `.codebuddy/skills/` 下的 SKILL.md Frontmatter

2.**项目内文档搜索**：使用搜索工具在项目中搜索 `.md` 文件中的相关关键词

3.**代码注释搜索**：搜索代码中的注释和命名获取设计意图

4.**历史上下文搜索**：使用历史上下文搜索工具查找历史对话中的相关信息

5.**外部知识检索**：当项目内信息不足时，可搜索外部知识库获取技术参考

#### 全新任务无文档覆盖时的处理

当 Agent 评估发现当前任务属于全新功能开发，且满足以下全部条件时：

- 扫描 Skill 目录后，没有任何现有文档的 description 能匹配当前任务
- 项目内也没有相关的设计文档或技术文档
- 该任务涉及新的子系统、新的协议、或新的模块间交互

则 Agent 应在方案输出中主动提议新建对应的 Skill 文档（遵循 §6.5 流程），说明：

- 建议的文档名称和存放位置
- 文档应覆盖的知识域
- 建议的章节大纲
- 与现有文档的关系

#### 搜索行为约束

- 搜索是 Agent 自主决策，无需询问用户许可（属于只读操作）
- 搜索过程不需要向用户展示（除非用户要求或搜索结果对方案决策有关键影响）
- 发现文档与代码不一致时，以代码为准开发，完成后提议更新文档
- 不为极其简单的独立修改执行全量文档扫描（如仅改一个常量值、修一个拼写错误）
- 搜索深度与任务复杂度成正比：简单任务快速扫描，复杂任务深入检索
